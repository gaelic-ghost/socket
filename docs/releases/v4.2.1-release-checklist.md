# `v4.2.1` Release Checklist

## Purpose

This checklist is the maintainer-facing candidate path for `v4.2.1`.

Use it when the release scope is the testing and release-workflow simplification that keeps one authoritative maintainer validation lane, moves CLI smoke checks into that lane, and makes `release-prepare.sh` a thinner branch-side handoff step.

For the standing branch-versus-main release contract, keep using [release-workflow.md](../maintainers/release-workflow.md). This document is only the candidate-specific checklist and release-shape note for `v4.2.1`.

## Proposed Release Number

Target the next release as `v4.2.1`.

That patch bump matches the current change shape:

- no public transport, embedding, or runtime API break
- maintainer and CI workflow simplification without changing the published server protocol
- release-prep cleanup that reduces duplicate staging work before merge
- docs alignment around the normal package lane, maintainer lane, and publish-time lane

## Pre-Tag Checklist

- keep the worktree clean before release preparation or publish
- confirm the authoritative maintainer validation path passes:

```bash
sh scripts/repo-maintenance/validate-all.sh
```

- run the live-service resident-model unload preflight before the live smoke suite, leaving the installed LaunchAgent service in place:

- run the current live end-to-end smoke gate in the repo-approved one-suite-at-a-time shape:

```bash
SPEAKSWIFTLYSERVER_E2E=1 xcrun swift test --filter ServerTransportE2ETests
```

## Suggested Release Notes Shape

Center the release notes on four points:

- the single authoritative maintainer validation lane
- the moved CLI smoke checks inside that lane
- the thinner branch-side `release-prepare.sh` role
- the clearer docs around development, validation, and publish-time release flow

## Publish Reminder

If this candidate moves forward, use the split release workflow rather than tagging directly from a feature branch:

1. run `scripts/repo-maintenance/release-prepare.sh --version v4.2.1` from the feature branch or release candidate worktree
2. let the PR merge back to `main`
3. fast-forward local `main`
4. run `scripts/repo-maintenance/release-publish.sh --version v4.2.1 --skip-live-service-refresh`
