# `v4.0.0-rc.1` Release Checklist

## Purpose

This checklist is the maintainer-facing candidate path for the next major prerelease after `v3.3.0`.

Use it when the release scope is the consumer-facing embedding cleanup that flattens the app-owned embedded runtime surface onto one observable object, removes accidental public API leakage, and updates the package docs to match that simpler ownership story.

For the standing branch-versus-main release contract, keep using [release-workflow.md](./release-workflow.md). This document is only the candidate-specific checklist and release-shape note for `v4.0.0-rc.1`.

## Proposed Release Number

Target the next release as `v4.0.0-rc.1`.

That major prerelease bump matches the current change shape:

- breaking replacement of the old embedded consumer model with the flattened `EmbeddedServer` observable owner
- breaking lifecycle and binding vocabulary shift from the older session-plus-state split to direct `liftoff()` and direct observable properties on the app-owned object
- intentional tightening of the published package surface so CLI-only and transport-local helpers are no longer presented as library API
- documentation realignment across README and DocC so the public package story matches the actual intended consumer path

## Pre-Tag Checklist

- keep the worktree clean before release preparation or publish
- confirm the default package lane still passes:

```bash
xcrun swift build
xcrun swift test
```

- confirm the maintainer validation path passes:

```bash
sh scripts/repo-maintenance/validate-all.sh
```

- run at least a small live end-to-end smoke pass across both transport families:

```bash
SPEAKSWIFTLYSERVER_E2E=1 xcrun swift test --filter 'http text profile lifecycle covers stored active effective and persistence flows'
SPEAKSWIFTLYSERVER_E2E=1 xcrun swift test --filter 'mcp catalog control resources prompts and subscriptions stay live and accurate'
```

- confirm the resolved `SpeakSwiftly` dependency is still current before tagging

## Suggested Release Notes Shape

Center the release notes on four points:

- the new `EmbeddedServer` app-consumer model
- the breaking lifecycle rename and flattening of observable properties and control actions onto that single object
- the package-surface cleanup that stops exporting accidental internal types
- the documentation refresh that now teaches the embedded path in the same shape the package actually expects consumers to use

## Publish Reminder

If this candidate moves forward, use the split release workflow rather than tagging directly from a feature branch:

1. run `scripts/repo-maintenance/release-prepare.sh --version v4.0.0-rc.1` from the release branch or worktree candidate
2. let the PR merge back to `main`
3. fast-forward local `main`
4. run `scripts/repo-maintenance/release-publish.sh --version v4.0.0-rc.1 --skip-live-service-refresh`
5. mark the GitHub release as a prerelease after publish because the current publish script does not set that flag directly
