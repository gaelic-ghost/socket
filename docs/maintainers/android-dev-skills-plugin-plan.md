# Android Dev Skills Plugin Plan

This plan records the first durable shape for a Socket-hosted Android development skills plugin.

The plugin's job is to help agents maintain Android applications and libraries with Kotlin-first guidance, Java interoperability, Java-only project support where a repository requires it, Gradle and Android Gradle Plugin alignment, emulator-aware validation handoffs, lint, tests, and release readiness.

## Intent

The `android-dev-skills` plugin should help agents do five things:

- choose the Android project, module, or maintenance shape before implementation starts
- preserve repo-local Kotlin, Java, Gradle, Android Gradle Plugin, Compose, XML view, and release decisions
- implement Android app or library changes without duplicating generic JVM backend guidance
- run and explain Android build, test, lint, packaging, emulator-aware validation, and release-readiness workflows
- hand off emulator operation and device debugging to the existing Android testing plugin instead of bundling duplicate runtime tooling

This is a companion guidance plugin, not a runtime plugin. The first version should not bundle Android Studio, Gradle, the Android SDK, an emulator controller, a device farm, a template feed, or machine-local SDK state.

## Packaging Direction

Package the guidance as an independent child plugin under:

```text
plugins/android-dev-skills/
```

The child plugin should own its Codex-facing guidance surface:

- `.codex-plugin/plugin.json`
- `skills/` once real skills land
- plugin metadata, skill metadata, `AGENTS.md`, or maintainer notes that explain the plugin's role
- any validation scripts needed for the plugin's own authored guidance

The root Socket marketplace should list `android-dev-skills` as `NOT_AVAILABLE` while it remains a placeholder. Switch the marketplace entry to `AVAILABLE` only after the first real skill content lands and root validation passes.

## Boundary With Server-Side JVM

`android-dev-skills` owns Android app and platform work:

- Android app, library, and multi-module project-shape discovery
- Android Gradle Plugin and build variants
- Kotlin-first Android implementation
- Java interoperability inside Android projects
- Java-only Android maintenance when repo defaults require it
- manifests, resources, permissions, signing, lint, tests, emulator-aware validation, and release readiness

`server-side-jvm` owns non-Android JVM backend work:

- Java, Scala, and future Clojure services
- shared non-Android JVM libraries
- backend Gradle, Maven, and SBT workflows
- service runtime configuration, persistence, observability, package, CI, and deployment-adjacent guidance

When a repository contains both an Android app and a JVM backend, route each change to the owning plugin. Shared Gradle mechanics do not make Android responsible for backend service design.

## Kotlin And Java Posture

Android guidance should be Kotlin-first because modern Android APIs and examples usually center Kotlin, but Java remains supported when the repository requires it.

That means:

- preserve existing repository language choices
- do not rewrite Java Android code to Kotlin just because Kotlin is preferred
- do not add Java-only Android patterns to Kotlin-first codebases unless interoperability requires it
- ask before large Kotlin/Java migration work
- explain Java interoperability explicitly when a Kotlin feature affects public APIs, annotations, nullability, SAM conversion, or generated bytecode behavior
- keep Java-only support clear for legacy, enterprise, generated, or repo-default Android projects

## Documentation Sources

Use official or canonical documentation first when authoring skills:

- Android Developers documentation
- Android Gradle Plugin and Gradle documentation
- Kotlin documentation for Android-facing language behavior
- Java documentation when Java-only or Java interop work matters
- Jetpack Compose, AndroidX, AppCompat, Room, Navigation, WorkManager, and other AndroidX docs when those libraries are in scope
- repository-selected testing, lint, CI, signing, release, and Play delivery documentation

When a skill relies on documentation, translate the relevant rule into practical workflow guidance. Do not drop citations into a skill as a substitute for explaining the effect on project layout, build variants, validation, release behavior, or user-facing diagnostics.

## Proposed Skill Inventory

### `android-dev:choose-project-shape`

Help an agent decide how Android work should fit into a user's repository before implementation starts.

This skill should classify the requested work:

- Android app
- Android library
- multi-module Android project
- Kotlin-first app
- Java-only app
- mixed Kotlin and Java app
- Compose UI
- XML view or AppCompat UI
- AndroidX library maintenance
- test, lint, signing, release, or dependency maintenance task

The output should recommend module ownership, language posture, Gradle and AGP checks, validation commands, Android testing handoffs, and documentation updates.

### `android-dev:gradle-agp-workflow`

Guide agents through Android Gradle and Android Gradle Plugin maintenance.

This skill should cover:

- detecting Gradle wrapper, AGP, Kotlin plugin, Java toolchain, and Android SDK expectations
- build variants, flavors, build types, signing config, and namespace/package alignment
- dependency version catalogs or repository-selected dependency management
- generated sources and build cache boundaries
- targeted assemble, test, lint, and dependency checks

### `android-dev:build-kotlin-android`

Guide agents through Kotlin-first Android implementation.

This skill should cover:

- activity, fragment, service, receiver, and app architecture ownership
- Compose and XML view boundaries according to repo defaults
- coroutines and lifecycle-aware work
- AndroidX and Jetpack library use
- nullability, state, persistence, and background work
- focused tests and lint validation

### `android-dev:java-android-workflow`

Guide agents through Java-only Android maintenance and Java interoperability inside Kotlin-first projects.

This skill should cover:

- Java Android app and library code paths
- Kotlin-to-Java and Java-to-Kotlin API boundaries
- annotations, nullability, SAM types, generated bytecode expectations, and AndroidX interop
- preserving Java codebases without involuntary Kotlin migration
- focused tests and lint validation

### `android-dev:testing-lint-workflow`

Run, debug, filter, and explain Android tests and lint.

This skill should cover:

- local unit tests
- instrumentation test handoffs
- Compose UI test handoffs
- lint configuration and baseline policy
- targeted Gradle tasks
- emulator-aware validation handoffs to the Android testing plugin
- readable failure summaries and next checks

### `android-dev:release-readiness-workflow`

Guide Android release-readiness checks without starting a publish workflow by default.

This skill should cover:

- version code and version name
- signing and keystore boundaries
- release build types
- ProGuard/R8 and mapping outputs
- app bundles and APKs
- Play delivery handoffs
- privacy, permissions, and changelog checks
- repository-owned release automation when publish work is explicitly requested
