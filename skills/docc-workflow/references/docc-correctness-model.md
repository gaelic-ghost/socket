# DocC Correctness Model

Use these three categories explicitly when reviewing DocC work:

## Content Correctness

Does the prose describe the code's real behavior, constraints, side effects, and intended usage?

## DocC Correctness

Does the content look structurally sound for DocC, including plausible symbol links, article boundaries, extension-file targeting, landing-page organization, and topic grouping?

## Project Correctness

Does the repository shape and project setup appear ready for the intended DocC generation or export path?

Phase one should own the first two strongly and only diagnose or hand off the third.
