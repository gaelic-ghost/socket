# `v3.1.3` Release Checklist

> Historical note: this checklist captures the `v3.1.3` release candidate process that existed before the repo adopted the split `release-prepare.sh` and `release-publish.sh` contract. For the current standing release workflow, use [release-workflow.md](./release-workflow.md).

## Purpose

This checklist is the maintainer-facing release path for the next patch after `v3.1.2`.

Use it to cut `v3.1.3` from the current hardening branch, with one explicit addition over the earlier patch flow: live E2E coverage should run before tagging, and the staged live-service path should get a real post-tag health verification pass before the release is considered complete.

## Proposed Release Number

Target the next release as `v3.1.3`.

That bump matches the current change shape:

- additive hardening for shared-host lifecycle ownership and startup sequencing
- additive operator-facing `healthcheck` and LaunchAgent promotion improvements
- additive embedded-session runtime controls for backend switching plus model reload and unload flows
- additive repo-maintenance and CI discipline around formatting, linting, and the `xcrun` toolchain lane
- additive dependency refresh to `SpeakSwiftly 3.0.5` and `TextForSpeech 0.17.0`
- no intentional public server-contract break in this pass

## Pre-Tag Checklist

- Keep the worktree clean before tagging. The release script still refuses to run from a dirty checkout.
- Decide the status of any unrelated local edits first. In the current checkout, `AGENTS.md` has been left intentionally outside the dependency-update commit and should not be swept into the release accidentally.
- Confirm the maintainer validation wrapper still passes:

```bash
sh scripts/repo-maintenance/validate-all.sh
```

- Confirm the normal package lane still passes with Xcode's selected toolchain:

```bash
xcrun swift package dump-package
xcrun swift build
xcrun swift test
```

- Confirm the operator smoke surfaces still render:

```bash
xcrun swift run SpeakSwiftlyServerTool help
xcrun swift run SpeakSwiftlyServerTool launch-agent print-plist
xcrun swift run SpeakSwiftlyServerTool healthcheck
```

- Run the opt-in live E2E suite before tagging:

```bash
SPEAKSWIFTLYSERVER_E2E=1 xcrun swift test
```

- If the full live E2E run is too costly to repeat during candidate iteration, at minimum record which subset was run, why the subset was chosen, and which lanes remain deferred. The preferred release path is still the full opt-in suite.

Current candidate note:

- the current `deps/speakswiftly-3-0-5` branch has already passed `SPEAKSWIFTLYSERVER_E2E=1 xcrun swift test` on 2026-04-16 after aligning the audible E2E helpers with `SpeakSwiftly 3.0.5`'s lazy playback-engine initialization behavior

## Tagging And Release Command

Use the repo-maintenance release flow for the actual tag:

```bash
scripts/repo-maintenance/release.sh --version v3.1.3 --skip-live-service-refresh
```

Use `--skip-live-service-refresh` only when you intentionally want tagging, artifact staging, and GitHub release creation without touching the live per-user service on the current machine.

If you want the tag flow to refresh the live LaunchAgent-backed service immediately, run:

```bash
scripts/repo-maintenance/release.sh --version v3.1.3
```

Or pass an explicit config path:

```bash
scripts/repo-maintenance/release.sh \
  --version v3.1.3 \
  --live-service-config-file "/absolute/path/to/server.yaml"
```

The script will:

- run repo-maintenance validation unless `--skip-validate` is passed
- build `SpeakSwiftlyServerTool` in release mode
- stage the release artifact under `.release-artifacts/v3.1.3`
- refresh `.release-artifacts/current`
- create the annotated `v3.1.3` tag
- push the branch and tag
- create the GitHub release object through `gh` when available
- optionally refresh the live LaunchAgent-backed service from the staged artifact

## Required Post-Tag Verification

If the release flow refreshed the live service, or if you manually install the staged artifact afterward, run the post-tag health verification pass before calling the release done:

```bash
.release-artifacts/current/SpeakSwiftlyServerTool healthcheck
```

Also verify the LaunchAgent-facing state explicitly:

```bash
.release-artifacts/current/SpeakSwiftlyServerTool launch-agent status
```

That post-tag verification should confirm:

- the staged artifact is the one the live LaunchAgent is actually pointing at
- `GET /healthz` passes
- `GET /runtime/host` passes
- MCP `initialize` against `/mcp` passes
- the intended config path matches the staged live-service install path

If the tagged release intentionally skips live-service refresh, record that plainly in the release notes or maintainer log so there is no ambiguity about what was and was not exercised.

## Suggested Release Notes Shape

Center the release notes on the additive hardening story:

- shared-host lifecycle ownership and startup truth improvements
- first-class `healthcheck` and stronger operator diagnostics
- embedded-session model lifecycle controls that now match the host HTTP and MCP surfaces more closely
- LaunchAgent `promote-live` and explicit executable-path handling

- runtime dependency refresh to `SpeakSwiftly 3.0.5` and `TextForSpeech 0.17.0`

Call out the carried-forward known issues separately:

- current `mlx-audio-swift` warning noise during build and test
- the still-open allocator-warning ownership audit
- the still-missing dedicated release helper for staged-artifact plus live-service verification, if it remains manual for this release

## Follow-Up After Release

- If the live E2E pass exposes failures, turn those into a dedicated maintainer note before retrying the release.
- If the live-service verification pass reveals any mismatch between staged artifact, LaunchAgent config path, and runtime host state, treat that as release-blocking for the accessibility-service install surface.
- If `v3.1.3` ships cleanly, consider closing or splitting the remaining release-verification follow-up in [`docs/maintainers/live-service-reliability-follow-ups.md`](../maintainers/live-service-reliability-follow-ups.md) so the roadmap no longer mixes shipped healthcheck work with the still-manual post-tag flow.
