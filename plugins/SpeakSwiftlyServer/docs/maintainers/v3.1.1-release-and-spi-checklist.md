# `v3.1.1` Release And SPI Submission Checklist

> Historical note: this checklist records the `v3.1.1` release and SPI submission flow before the repo adopted the split `release-prepare.sh` and `release-publish.sh` contract. For the current standing release workflow, use [release-workflow.md](./release-workflow.md).

## Purpose

This checklist turns the current runtime, validation, and package-discovery work into one concrete next-step plan for the next patch release and the first Swift Package Index submission.

Treat this as the maintainer-facing path for the current patch release. It is not a permanent replacement for the repo-maintenance release script, but it is the current source of truth for how to use that script safely for this package and how to follow it with the SPI submission flow.

## Proposed Release Number

Target the next release as `v3.1.1`.

That bump matches the current change shape:

- additive runtime profile-root override support across embedded, foreground, and LaunchAgent entrypaths
- additive embedded-session concurrency cleanup so app-owned startup no longer runs through the main actor
- additive maintainer notes and release-readiness validation for the new embedding surfaces
- no intentional breaking package-surface change in this pass

## Pre-Tag Checklist

- Keep the worktree clean before tagging. The release script already refuses to run from a dirty checkout.
- Confirm the package still passes the maintainer validation wrapper:

```bash
sh scripts/repo-maintenance/validate-all.sh
```

- Confirm the normal package path still passes with Xcode's selected toolchain:

```bash
xcrun swift package dump-package
xcrun swift build
xcrun swift test
```

- Confirm the operator smoke surfaces still render:

```bash
xcrun swift run SpeakSwiftlyServerTool help
xcrun swift run SpeakSwiftlyServerTool launch-agent print-plist
```

- Decide whether the opt-in live E2E suites should run for this release candidate. They are still gated behind `SPEAKSWIFTLYSERVER_E2E=1`, so they are not part of the default maintainer validation path yet.

## Tagging And Release Command

Use the repo-maintenance release flow for the actual tag:

```bash
scripts/repo-maintenance/release.sh --version v3.1.1 --skip-live-service-refresh
```

Use `--skip-live-service-refresh` by default unless you explicitly want this release pass to promote the staged artifact into the live LaunchAgent-backed service on this machine.

The script will:

- run repo-maintenance validation unless told not to
- build `SpeakSwiftlyServerTool` in `release` mode
- stage the release artifact under `.release-artifacts/v3.1.1`
- refresh `.release-artifacts/current`
- create the annotated `v3.1.1` tag
- push the branch and tag
- create the GitHub release object through `gh` when available

If you intentionally want the release flow to refresh the live per-user service afterward, rerun without `--skip-live-service-refresh` or pass an explicit config path with `--live-service-config-file /absolute/path/to/server.yaml`.

## Swift Package Index Readiness Check

Before submitting, confirm the package still matches the current public SPI requirements:

- repository is public
- `Package.swift` is valid at the repository root
- the package has at least one semantic-version tag
- `swift package dump-package` produces valid JSON
- the package compiles successfully
- `.spi.yml` is present and still intentionally minimal for the `SpeakSwiftlyServer` documentation target

Current official references:

- [Add a Package](https://swiftpackageindex.com/add-a-package)
- [FAQ](https://swiftpackageindex.com/faq)
- [SPIManifest documentation](https://swiftpackageindex.com/swiftpackageindex/spimanifest/documentation/spimanifest/commonusecases)

## SPI Submission Steps

Once `v3.1.1` is pushed and the GitHub release exists:

1. Submit the repository through the Swift Package Index add-package flow using the repository URL with the `.git` suffix:
   `https://github.com/gaelic-ghost/SpeakSwiftlyServer.git`
2. Wait for the package page and the first build results to appear.
3. Verify that the hosted documentation build picks up the `SpeakSwiftlyServer` DocC catalog cleanly.
4. Review the rendered package page for README presentation, release notes, compatibility results, and hosted-doc navigation.
5. Once the package page exists, use the maintainership flow on Swift Package Index to claim the package and then add the generated compatibility badges back into `README.md` if they look worth keeping.

## Post-Submission Follow-Through

- If SPI rendering exposes awkward DocC navigation or article overlap, tighten the catalog after seeing the hosted output instead of guessing up front.
- If SPI build logs surface package-specific warnings beyond the current known `mlx-audio-swift` noise, capture those in a follow-up maintainer note before the next release.
- If the package lands cleanly, close the Milestone 10 submission item in `ROADMAP.md`.
