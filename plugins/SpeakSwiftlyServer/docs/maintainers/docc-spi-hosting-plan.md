# DocC And SPI Hosted Docs Plan

## Purpose

This document captures the first deliberate pass for getting `SpeakSwiftlyServer` onto Swift Package Index with hosted documentation that feels intentional instead of auto-generated by accident.

The immediate goal is not a perfect documentation portal. The immediate goal is to give SPI enough stable DocC structure that the hosted docs page explains the public library surface clearly, matches the repository's current package story, and can grow without another rewrite.

## Current State

- The package now includes a minimal root `.spi.yml` that asks Swift Package Index to build documentation for the `SpeakSwiftlyServer` library target.
- The package now includes a first `.docc` catalog at `Sources/SpeakSwiftlyServer/SpeakSwiftlyServer.docc`.
- The baseline library-first article set is in place:
  - `SpeakSwiftlyServer.md`
  - `Articles/Embedding-The-Server.md`
  - `Articles/App-Managed-Install-Layout.md`
  - `Articles/Operator-Surfaces.md`
- The highest-value public embedding and snapshot surfaces now have an initial symbol-documentation pass, so the hosted docs should no longer be purely symbol-heavy and context-light.
- The executable workflow (`SpeakSwiftlyServerTool`) is still important to operators, but the current SPI-hosted documentation scope remains intentionally centered on the `SpeakSwiftlyServer` library product because that is what SPI can document most naturally.
- The package now also carries a deliberately small executable companion lane for hosted docs:
  - `Articles/Using-The-Command-Line-Tool.md`
  - `Articles/LaunchAgent-Workflow.md`
  This companion set is intentionally narrative and operator-oriented without trying to duplicate the full repository README or API reference.

## Phase 1: First DocC Catalog

Status: landed.

The initial DocC catalog now lives under:

The baseline files now in place are:

  The package landing page. Explain what the library is for, the difference between the library and the executable, and the main public entrypoints.

- `Articles/Operator-Surfaces.md`

## Phase 2: Symbol Documentation Pass

Status: initial pass landed, follow-up tightening still possible.

Add concise DocC comments to the highest-value public symbols before expecting the hosted docs to feel good:

- `EmbeddedServerSession`
- `ServerState`
- `ServerRuntimeEntrypoint`
- `ServerInstallLayout`
- `ServerInstalledLogs`
- The public snapshot types in `HostStateModels.swift`, `QueueStatusModels.swift`, `JobEventModels.swift`, and `ProfileModels.swift`
- `SpeakSwiftlyServerToolCommand` and related public command-error types if we want the generated symbol docs to cover CLI-facing types cleanly

The goal here is not to narrate every stored property. The goal is to make the generated symbol pages answer:

- what this type represents
- who should use it
- whether it is app-facing, operator-facing, or transport-facing
- what adjacent type a reader should look at next

## CI Follow-Through

Status: baseline build check landed locally and in CI.

The package CI and maintainer validation path should help catch regressions that would make SPI-hosted docs or package presentation look broken.

Immediate CI expectations:

- keep `xcrun swift package dump-package`
- keep `xcrun swift build`
- keep `xcrun swift test`
- keep `xcrun swift package --allow-writing-to-directory .build/docc generate-documentation --target SpeakSwiftlyServer --disable-indexing --output-path .build/docc`
- keep `xcrun swift run SpeakSwiftlyServerTool help`
- add `xcrun swift run SpeakSwiftlyServerTool launch-agent print-plist`

## Submission Order

1. Keep `.spi.yml` intentionally minimal while the first SPI-hosted docs pass stays library-first.
2. Land the first `.docc` catalog and symbol-doc pass.
3. Add any CI or local build check needed to keep docs generation honest.
4. Add the short tutorial-style walkthrough if the hosted docs still need a clearer first-use path.
5. Submit the package to Swift Package Index.
6. Review the resulting hosted docs and compatibility pages, then tighten wording or navigation based on what SPI actually renders.
