# Style Rules

- Keep accessibility guidance direct, concrete, and repo-grounded.
- Treat `ACCESSIBILITY.md` as a project control document, not as a public marketing statement or a legal certification artifact.
- Prefer short explanatory paragraphs plus small bullet lists when they improve scanability.
- Keep `Standards Baseline` precise about target standards and claims language.
- Keep `Accessibility Architecture` focused on real semantics, input behavior, focus, announcements, contrast, motion, zoom, and alternatives.
- Keep `Engineering Workflow` focused on implementation rules, testing expectations, and merge-quality requirements.
- Keep `Known Gaps` explicit and honest rather than implying the project has no limitations by default.
- Prefer fenced code blocks with language info strings when showing automation or verification commands.
